# DDD Spec Generator

Generate structured behavior contracts (specs) for each feature area in a DDD project roadmap. Specs sit between roadmap (what to build) and develop (how to build), defining acceptance criteria, data models, API contracts, and boundary conditions.

**Core purpose:** Prevent direction drift by anchoring ddd-develop plans to numbered acceptance criteria (AC-1, AC-2...) and unified data models.

Supports three input modes:
1. **Single** — `/ddd-spec P0.1` generates a spec for one feature area
2. **Range/Phase** — `/ddd-spec P0.1 - P0.3` or `/ddd-spec P0` generates specs for multiple feature areas
3. **Batch (from roadmap)** — triggered by ddd-roadmap after roadmap generation, covers all feature areas

**Announce at start:**
- If arguments provided: "Using ddd-spec to generate behavior contracts for [scope]."
- If batch from roadmap: "Using ddd-spec to generate behavior contracts for all feature areas."

## Step 0: Classify Input

Parse arguments to determine scope:

- `P{n}.{m}` — single feature area (e.g., P0.1)
- `P{n}.{m} - P{n}.{k}` — range of feature areas
- `P{n}` — entire phase (all feature areas in that phase)
- No args — prompt user to select scope

**Parsing rules:**
- Scope tokens are `P` followed by digits and dots: `P[0-3]`, `P[0-3].[1-9]`
- Ranges use ` - ` (space-hyphen-space) between two scope tokens
- Commas or spaces separate enumerated items

**Validation:** Check that scope tokens reference feature areas that exist in the roadmap. If a token doesn't match any heading in `docs/roadmap/P[0-3]-*.md`, report an error and suggest valid options.

---

## Step 1: Read Context

For each feature area in scope, gather:

1. **Roadmap section** — Read the corresponding phase document, extract the feature area's heading, sub-feature headings, and all checkbox items. This is the primary input for spec generation.
2. **Project context** — Read CLAUDE.md for:
   - DDD layer structure (which directories map to Domain, Application, Infrastructure, Presentation)
   - Tech stack (language, framework, database, test framework)
   - Architectural conventions and patterns
3. **Existing code** — If relevant modules already exist, scan for:
   - Current data models (entity definitions, schemas, types)
   - Existing API patterns (route definitions, response formats)
   - Test patterns (test framework, assertion style)
4. **Existing specs** — Check `docs/specs/` for already-approved specs in the same phase. Pass these to subagents for cross-area consistency (shared entity references, consistent API patterns).

**Skip logic:** If a spec file already exists for a feature area with `status: approved`, skip it and inform the user:

```
Skipping P0.1 — approved spec already exists at docs/specs/P0.1-user-authentication.md
```

---

## Step 2: Dispatch Subagents

For each feature area that needs a spec, dispatch an independent Agent subagent. This prevents attention degradation when processing multiple feature areas.

**Why subagents:** Each feature area's spec requires focused reasoning about data models, acceptance criteria, and boundary conditions. A single session generating 5+ specs will produce declining quality for later specs. Isolated subagents maintain full attention per spec.

**If only one feature area in scope:** Skip subagent dispatch, generate the spec directly in the current session.

**If multiple feature areas:** Dispatch one Agent per feature area with this prompt:

```
You are generating a structured behavior contract (spec) for a DDD project feature area.

## Feature Area
[Phase and feature area ID, e.g., P0.1]
[Feature area title from roadmap heading]

## Roadmap Items
[Full text of the feature area section from roadmap, including sub-features and checkbox items]

## Project Context
- **Tech Stack:** [from CLAUDE.md]
- **DDD Layers:** [directory → layer mapping from CLAUDE.md]
- **Conventions:** [relevant patterns from CLAUDE.md]

## Existing Code Context
[Current data models, API patterns, test patterns found in Step 1 — or "Greenfield, no existing code" if none]

## Related Specs (for consistency)
[Summaries of approved specs in the same phase, focusing on shared entities and API patterns — or "None" if first spec]

## Step 3: Write Spec Files

For each generated spec:

1. Create directory `docs/specs/` if it doesn't exist
2. Write to `docs/specs/P{phase}.{area}-{slug}.md`

   - Example: `docs/specs/P0.1-user-authentication.md`
3. Status is set to `draft` in the frontmatter

**On change request:** Edit the spec file with requested changes, then re-present for review.

## Phase 1.5: SPEC GATE

Before generating a plan, verify that a behavior contract (spec) exists for this feature area. The spec anchors the plan to explicit acceptance criteria, preventing direction drift.

**Skip this gate if:**
- `source = "ad-hoc"` AND user passed `--skip-spec` flag
- The development target is a bug fix or refactoring task (not a feature)

### Gate Logic

1. **Determine feature area** — Extract `P{phase}.{area}` from the development target:
   - Roadmap item `P0.1.2` → feature area `P0.1`
   - Roadmap item `P0.2.1` → feature area `P0.2`
   - Ad-hoc requirement → attempt to match against existing roadmap feature areas; if no match, skip gate

2. **Find spec file** — Search for `docs/specs/P{phase}.{area}-*.md` using Glob

3. **Check spec status:**
   - **File not found** → BLOCK:
     ```
     Spec not found for feature area P0.1.
     
     A behavior contract is required before development to prevent direction drift.
     Run: /ddd-spec P0.1
     Or pass --skip-spec to bypass this gate for ad-hoc work.
     ```
     Wait for user response. Do NOT proceed without spec or explicit bypass.
   
   - **File found, status: draft** → BLOCK:
     ```
     Spec for P0.1 exists but is still in draft status.
     
     File: docs/specs/P0.1-user-authentication.md
     Please review and approve the spec before development.
     ```
     Wait for user response.
   
   - **File found, status: approved** → PASS. Read the spec file and pass its content to Phase 2.

### Spec Context for Phase 2

When the gate passes, extract and carry forward:
- **Acceptance criteria** relevant to the current item (from the Coverage table)
- **Data model definitions** (all models in the spec — shared across items)
